下面是按阶段拆解、可逐步实施且每一步都可独立调试验证的实现计划。每个阶段包含实施要点、交付物与验证方法，确保你可以逐步落地弹性伸缩能力。

整体路径

- 先搭建“服务注册/发现 + 健康检测”的基础能力，再实现 Agent 的后端路由与 Game 组内分片。
- 完成基础路由后，加入 Actor 归属与迁移闭环，最后补齐观测与自动扩缩容。
- 每一步都以“最小可用”目标交付，避免大爆炸式集成。
阶段 1：服务注册与健康检测

- 实施要点
  - 在 ggs/app/{agent,game,login,platform} 启动时注册自身到 etcd ，写入键： /ggs/services/<service>/<instanceId> -> {addr, meta, ttl} 。
  - 使用租约与 keepalive 维持存在；下线时释放租约；异常停止由 TTL 自动过期。
  - 暴露 HTTP /healthz （返回 200 OK 简单 JSON），用于容器编排与 LB健康检查。
- 交付物
  - ServiceRegistry 小库（ internal/core/cluster/registry 或类似路径）。
  - 每个服务的 Register/Unregister 接口与 healthz 端点。
- 验证
  - 启动两个 agent 、两个 game 实例， etcdctl get /ggs/services/... --prefix 能看到动态变更。
  - 停止其中一个实例，键在 TTL 时间后自动消失。
  - curl http://127.0.0.1:<port>/healthz 返回 200 。
阶段 2：Agent 后端路由（按 serverId）

- 实施要点
  - 在 agent 内维护 serverId -> game 组实例列表 映射（从 etcd watch）。
  - 引入一致性哈希或 Rendezvous Hash（依赖已有 github.com/dgryski/go-rendezvous ），对所有请求先分配到目标逻辑服，再选组内 worker。
  - 防御性重试：选定 worker 不可达时，快速重试同组健康 worker。
- 交付物
  - ggs/app/agent/internal/router 路由器实现，含 Watch 逻辑和 Hash 分配。
- 验证
  - 启动 1 agent + 2 game，同一 serverId 请求均路由到对应 game 组，且分布基本均衡。
  - 停掉一个 game 实例，请求自动转向健康实例。
  - 用简单压力脚本验证分布与延时（QPS、均匀性）。
阶段 3：Game 组内分片（无迁移）

- 实施要点
  - 将单逻辑服定义为“Game 组”，组内多个 worker 节点服务该服的玩家与场景。
  - 先实现“无状态分片”：对 uid/characterId 做一致性哈希，落到某个 worker；worker 本地构造 Session/Actor。
  - 暂不迁移，只确保路由到同一 worker（粘性）。
- 交付物
  - ggs/app/game/internal/router ，组内分片策略配置（权重、健康）。
- 验证
  - 启动 2 worker，同一玩家的所有请求始终落到同一 worker。
  - 动态增减 worker，验证新玩家按新环分布，老玩家不受影响。
阶段 4：Actor 归属注册表

- 实施要点
  - 建立 ActorRegistry ： playerId -> workerId ，使用 redis 或 etcd 存储并设置 TTL，worker 定期续租。
  - Agent 在转发前查询归属；worker 收到请求时校验归属一致，若不一致返回重定向或 Agent 更新一次归属缓存。
- 交付物
  - internal/core/actor/registry 小库； GetOwner/SetOwner/RefreshOwner 接口。
- 验证
  - 手动 SetOwner(playerId,w1) 后，路由一致；修改为 w2 后，新请求切换到 w2 （具备明确控制与可视化）。
阶段 5：最小迁移闭环（单 Actor）

- 实施要点
  - 为关键 Actor（如玩家）实现 Serialize/Deserialize 快照方法。
  - 迁移流程：源 worker 生成快照 → 目标 worker 加载 → 更新归属 → 源 worker 停止该 Actor。
  - 迁移窗口内对该 Actor 的新请求暂存或返回“稍后重试”，避免并发写入。
- 交付物
  - internal/core/actor/migration 模块，方法： MigrateActor(playerId, toWorkerId) 。
- 验证
  - 启动 2 worker，触发单玩家迁移；迁移后新请求在 toWorkerId 响应且状态保持一致。
  - 注入故障（目标加载失败）验证回滚与告警。
阶段 6：背压与可靠性

- 实施要点
  - 在 agent 与 game 引入限流与队列背压（队列满立即拒绝）。
  - agent 对满载的 game 返回标准错误码（如 ServerBusy ），客户端做指数退避重试。
  - 请求幂等：关键写操作增加 seq/nonce 去重。
- 交付物
  - 简单工作池或直接用现有 gutils/worker ；统一错误码定义在 ggs/internal/errs 。
- 验证
  - 压测触发背压，观察拒绝率与平均延时；确认系统不出现级联阻塞。
阶段 7：观测与告警

- 实施要点
  - 集成 Prometheus 指标：QPS、延时、连接数、迁移成功率、队列长度、GC/内存等。
  - 日志加 TraceID（在 gactor 请求上下文里植入），关键流量路径打点。
  - 可选：OpenTelemetry 简易链路追踪。
- 交付物
  - internal/libs/logger 扩展； metrics 暴露 /metrics 。
- 验证
  - Prometheus 成功拉取指标；本地 Grafana 可视化，设告警阈值（如迁移失败率升高）。
阶段 8：自动扩缩容控制器（先半自动）

- 实施要点
  - 控制器基于指标（延时、负载、Actor 数）决定扩容/缩容。
  - 先实现“半自动”：控制器仅生成建议与操作事件，人工确认执行；稳定后改自动。
  - 扩容：新增 worker 并加入哈希环；缩容：标记 draining ，停止分配新 Actor，迁移完成后下线。
- 交付物
  - internal/tools/autoscaler 或平台内置控制器；操作事件日志。
- 验证
  - 手动提升负载触发扩容建议；确认执行后负载降低。
  - 缩容流程完成且业务不中断。
阶段 9：灰度发布与滚动升级

- 实施要点
  - 新版本 game worker 引入低权重，逐步放量；监控稳定后提升权重。
  - 回滚策略：一键移除新版本节点并恢复旧版本，迁移必要的 Actor。
- 交付物
  - 部署策略配置；权重管理接口（ SetWeight(workerId, weight) ）。
- 验证
  - 灰度发布实验：观察错误率与延时，权重调整带来负载变化；回滚实验成功。
阶段 10：平台集成与管理

- 实施要点
  - platform 提供：创建逻辑服、绑定 serverId 、查看/调整 worker 权重、触发迁移与扩缩容等接口。
  - 管理面写操作加审计日志与幂等保障。
- 交付物
  - platform 的 REST 接口与前端管理页（可先纯后端）。
- 验证
  - 通过 platform 发起扩缩容与迁移操作，链路打通；回显状态与审计记录完整。
每阶段独立验证的本地运行建议

- 依赖启动
  - etcd 与 redis 可用 Docker 本地启动，或用现成环境。
- 运行单服务或组合服务
  - 登录服务：压测令牌签发与角色管理接口。
  - Agent+Game：本地起多实例，模拟客户端通过 Agent 发包，检查分片与迁移。
- 命令行/Make 目标
  - 为每阶段新能力添加 make 目标，便于快速验证与回归（例如 make run_agent , make run_game_w1 , make run_game_w2 , make migrate_player ）。
