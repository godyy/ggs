弹性伸缩实现计划（v2，基于当前代码分析）

概览

- 目标：以“最小可用”迭代落地，先打稳注册/探针与路由，再推进分片、归属、迁移与观测，最终实现半自动扩缩容与灰度能力。
- 当前状态要点：
  - 已有 etcd 集群发现与租约（internal/modules/cluster），并在 agent/game 集成。
  - 探针库（internal/libs/probe）已具备初始化封印、就绪策略/TTL/超时、/healthz 与 /readyz、Gin 与 net/http 适配；game 已接入 Mongo/Redis 检查。
  - login/platform 的 HTTP 服务基于 Gin，尚未接入探针；观测/自动扩缩容尚未落地。

实施路径（分阶段）

阶段 0：基础完善（小步快跑，1–2 天）

- 实施要点
  - 统一探针：在 login、platform 的 `startHttp()` 中接入 `probe.Init(...)` 与 `probe.RegisterGin(engine, "/healthz", "/readyz")`；所有服务统一 `/healthz`（存活）与 `/readyz`（业务就绪）。
  - 管理端口：建议为各服务提供管理 `ServeMux`（或复用当前），挂载 `/debug/pprof/*` 与探针，绑定内网或独立端口。
  - 键空间规范：按服务类型设置不同的 `CenterConfig.Root`（如 `/ggs/cluster/agent`、`/ggs/cluster/game`），避免混淆。
- 交付物
  - login/platform 探针接入补丁；管理端口示例；Root 命名策略文档。
- 验证
  - `curl /healthz` 返回 200；`curl /readyz` 在 `SetReady(true)` 后返回 200；依赖故障时 `/readyz` 返回 503 并列出失败项。

阶段 1：服务注册与健康检测（已具备约 70%，补齐 30%）

- 实施要点
  - 复用 `internal/modules/cluster` 完成 agent、game、login、platform 的 etcd 注册与 keepalive；统一 Root 和节点 ID 命名策略。
  - 探针完善：所有服务使用 `/healthz`（无外部依赖）与 `/readyz`（门闩 + 检查器），就绪后 `SetReady(true)`。
- 交付物
  - 各服务的注册/注销流程；探针路由统一实现。
- 验证
  - `etcdctl get <Root> --prefix` 可见在线节点；停止实例后键在 TTL 后消失；watch 收到新增/删除事件。

阶段 2：Agent 后端路由（按 serverId）

- 实施要点
  - 路由表：在 agent 维护 `serverId -> game组` 映射（从 `/ggs/cluster/game` watch 在线节点）；初期可用“默认组”方式，后续支持多逻辑服。
  - 分配算法：用 Rendezvous 哈希（go.mod 已引入）按 `serverId` 选择目标 game 节点；不可达时快速重试同组健康节点。
- 交付物
  - `app/agent/internal/router` 包，含 watch 与分配逻辑；均衡性与容错的单元测试。
- 验证
  - 单 agent + 多 game：同一 `serverId` 路由稳定且分布均衡；宕机时重试成功。

阶段 3：Game 组内分片（先粘性，无迁移）

- 实施要点
  - 分片策略：以 `uid/characterId` 做一致性哈希，粘性落到某个 worker；worker 本地构造 Session/Actor。
  - 暂不迁移，只确保同一玩家稳定落在同一 worker。
- 交付物
  - `app/game/internal/router`，实现粘性分片；支持权重与健康配置。
- 验证
  - 动态增减 worker，新玩家按新环分布，老玩家不受影响。

阶段 4：Actor 归属注册表（与 Meta 协同）

- 实施要点
  - `ActorRegistry`：键 `playerId -> workerId` 存储在 Redis（加 TTL + 续租），与 `MetaDriver` 区分职责。
  - 路由校验：Agent 转发前查询归属；worker 收到请求校验归属一致，必要时返回重定向或由 Agent 更新缓存。
- 交付物
  - `internal/modules/actor/registry` 小库，`GetOwner/SetOwner/RefreshOwner` 接口。
- 验证
  - 设置归属后路由一致；修改归属后新请求切换并可视化。

阶段 5：最小迁移闭环（单 Actor）

- 实施要点
  - 为关键 Actor（如玩家）实现 `Serialize/Deserialize` 快照接口。
  - 迁移流程：源 worker 生成快照 → 目标 worker 加载 → 更新归属 → 源 worker 停止该 Actor；窗口内对该 Actor 的新请求暂存或返回“稍后重试”。
- 交付物
  - `internal/modules/actor/migration` 模块，支持回滚与告警。
- 验证
  - 触发单玩家迁移；迁移后新请求在目标 worker 响应且状态一致；目标加载失败时回滚成功。

阶段 6：背压与可靠性

- 实施要点
  - 在 agent/game 引入工作池与队列满立即拒绝；统一错误码 `internal/base/errs`。
  - 客户端对 `ServerBusy` 做指数退避重试；关键写操作增加 `seq/nonce` 幂等去重。
- 交付物
  - 背压实现与错误码；压力测试脚本。
- 验证
  - 压测触发背压，观察拒绝率与延时；确认无级联阻塞。

阶段 7：观测与告警

- 实施要点
  - 集成 Prometheus 指标，暴露 `/metrics`（QPS、延时、连接数、迁移成功率、队列长度、GC/内存等）。
  - 日志统一 TraceID（在 gactor 上下文植入），关键路径打点；可选接入 OpenTelemetry。
- 交付物
  - `internal/libs/metrics` 新库；平台或服务侧指标页。
- 验证
  - Prometheus 成功拉取；Grafana 可视化；设告警阈值（如迁移失败率）。

阶段 8：自动扩缩容控制器（先半自动）

- 实施要点
  - 控制器周期采样指标，根据延时/负载/Actor 数生成扩缩容建议与操作事件（人工确认）；稳定后切自动。
  - 扩容：新增 worker 加入哈希环；缩容：标记 `draining`，停止新分配，迁移完成后下线。
- 交付物
  - `internal/tools/autoscaler` 或 platform 控制器；操作事件日志。
- 验证
  - 提升负载触发扩容建议；执行后负载降低；缩容流程无中断。

阶段 9：灰度发布与滚动升级

- 实施要点
  - 新版本 worker 以低权重进入环，逐步放量；稳定后提升权重。
  - 回滚：移除新版本节点，恢复旧版本；必要时迁移 Actor。
- 交付物
  - 权重管理接口 `SetWeight(workerId, weight)`；部署策略配置。
- 验证
  - 观测错误率与延时；权重调整带来负载变化；回滚成功。

阶段 10：平台集成与管理

- 实施要点
  - platform 提供：创建逻辑服、绑定 `serverId`、查看/调整 worker 权重、触发迁移与扩缩容；写操作审计与幂等保障。
- 交付物
  - platform 的 REST 接口与（可选）前端管理页。
- 验证
  - 通过 platform 发起扩缩容与迁移操作，链路打通；状态回显与审计完整。

验证清单（统一）

- 探针与管理端口
  - `/healthz` 返回 200；`/readyz` 在 `SetReady(true)` 后返回 200；Mongo/Redis 宕机时 `/readyz` 返回 503 且包含 `fails`。
  - pprof `/debug/pprof/*` 仅内网或独立端口访问。
- 集群注册
  - `etcdctl get <Root> --prefix` 可见在线节点；停实例后键在 TTL 后消失；watch 收到事件。
- 路由与分片
  - 单 agent + 多 game：同一 `serverId` 路由稳定且均衡；宕机时重试成功。
  - 组内分片：同一玩家稳定落在同一 worker；扩容后新玩家按新环分布。
- 归属与迁移
  - `ActorRegistry` 设置后路由按归属生效；迁移流程成功并可回滚。
- 观测与扩缩容
  - Prometheus 拉取 `/metrics`；压力测试触发半自动扩容建议，执行后负载降低；缩容流程无中断。

代码放置与集成建议

- 探针库：`internal/libs/probe`（已存在），继续保持轻量；所有服务统一接入。
- 注册/发现：复用 `internal/modules/cluster`；按服务类型设置不同 `Root`。
- Agent 路由：`app/agent/internal/router`；Game 分片：`app/game/internal/router`。
- Actor 归属/迁移：`internal/modules/actor/registry` 与 `internal/modules/actor/migration`。
- 观测：新增 `internal/libs/metrics`，统一封装 Prometheus 注册与 `/metrics` 路由。
- 自动扩缩容：`internal/tools/autoscaler` 或 platform 内置控制器。

安全与可用性建议

- pprof 与管理端口建议绑定内网或独立端口；生产环境加鉴权或网关白名单。
- `/healthz` 仅用于存活；`/readyz` 反映业务能力（门闩 + 检查器）。
- 就绪策略：初始化较长且依赖稳定用 `GateOnly`；依赖可能抖动用 `AlwaysCheck`；大集群降压用 `Cached`（TTL 建议 5–30s）。
- 检查器实现需遵守 `ctx` 超时/取消，避免 goroutine 泄漏；默认超时建议 ≥1s，依赖偏慢取 3–5s。

下一步建议（可直接落地补丁）

- 在 login 与 platform 接入 `probe`，统一 `/healthz` 与 `/readyz`，策略按依赖选择（login 依赖 Mongo/Redis 可用 `Cached`，platform 先 `GateOnly`）。
- 在 agent 落地最小后端路由骨架（watch `/ggs/cluster/game` + Rendezvous 分配），并提供均衡性/容错测试用例。
